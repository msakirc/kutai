## 1. Does

KDV is the **cloud LLM provider capacity tracker** — a sync, in-process state machine that
answers "can I call this cloud model right now?" and "how long until I can?" It is the cloud
counterpart to DaLLaMa (local). Concretely, from the code it:

- **Two-tier rate limiting**: per-model AND per-provider-aggregate RPM/TPM sliding windows
  (`rate_limiter.py:583` `RateLimitManager` holds `model_limits` + `_provider_limits`; both
  must pass `has_capacity` — `rate_limiter.py:637-650`). Provider aggregate is only created
  when the host passes aggregate values; if both are `None` the entry is skipped and per-model
  buckets gate alone (`rate_limiter.py:627-635`).
- **Daily/quota axes**: tracks RPD and TPD (plus a large set of forward-compat axis stubs —
  rph/rpw/rpmonth/tph/tpw/tpmonth/itpm/itpd/otpm/otpd/cpd/cpmonth at `rate_limiter.py:51-91`).
  RPD is decremented **locally on every attempt** (`record_request` → `_tick_rpd`,
  `rate_limiter.py:755-775`) so providers that send no daily headers (Gemini free tier) still
  surface exhaustion before the first 429. Daily reset is anchored to **next UTC midnight**
  (`_next_utc_midnight`, `rate_limiter.py:739-753`), not rolling-24h.
- **Header-derived live limits**: parses provider response headers into a `RateLimitSnapshot`
  and overrides configured defaults (`header_parser.py:130`, applied via
  `update_from_snapshot`, `rate_limiter.py:507`). Strips litellm's `llm_provider-` key prefix
  (`header_parser.py:151-157`).
- **429-body parsing**: `parse_429_body` extracts daily-axis quota from a Gemini
  RESOURCE_EXHAUSTED error body when headers don't carry it (`header_parser.py:79-127`).
- **`Retry-After` honoring**: parsed centrally (RFC 7231 delta-seconds or HTTP-date,
  `header_parser.py:178-203`); installs an rpm-reset floor with `remaining=0`
  (`rate_limiter.py:576-580`).
- **Adaptive 429 reduction + restore**: each 429 reduces RPM/TPM by 20% (floor 50% of
  original); restores 10%/cycle after 10 min clean (`record_429` `rate_limiter.py:450`,
  `maybe_restore_limits` `rate_limiter.py:477`).
- **Post-429 cooldown**: forces `rpm_remaining = 0` for 60 s after a 429 even if the sliding
  window decayed, to stop re-bursting onto the wall (`POST_429_COOLDOWN_SECONDS`,
  `rate_limiter.py:216-263`).
- **Per-provider circuit breaker**: sliding-window failure count, trip → cooldown, reset on
  success (`circuit_breaker.py:10`). Default 3 failures / 300 s window / 600 s cooldown
  (`config.py:11-13`).
- **Canary gate**: under uncertainty (boot, day-reset, breaker-reset, or any failure) only ONE
  call ("canary") is admitted per provider; others get `reason="canary_in_flight"` until it
  returns 200 (`kdv.py:283-296`, flipped at `post_call` `kdv.py:421-422` / re-locked on any
  failure `kdv.py:609-610`).
- **Provisional TPM reservation (admission-race close)**: `record_attempt` reserves estimated
  tokens + bumps RPM **before** the POST; `post_call` corrects to actual via a delta;
  `release_reservation` rolls back on failure (`kdv.py:152-223`).
- **Reliability scoring**: rolling 30-entry / 24 h per-model outcome window
  (`recent_success_rate`, `kdv.py:437`) + a cross-sibling `provider_prior_rate` for cold-start
  fill (`kdv.py:456`). Feeds nerd_herd S10.
- **Capacity-change callback**: fires `on_capacity_change` on `capacity_restored`, `limit_hit`,
  `circuit_breaker_tripped`, `circuit_breaker_reset` (`kdv.py:83`, `_fire`).
- **In-flight tracking + push to nerd_herd**: `begin_call`/`end_call` track dispatched-but-
  unconfirmed calls with TTL pruning, and (when wired) push a `CloudProviderState` carrying
  `in_flight` counts into nerd_herd for pool-pressure (`in_flight.py:34`, `__init__.py:43-71`).
- **State persistence**: `snapshot_state`/`restore_state` round-trip adapted limits, daily
  counters, breaker state, enabled-at, counters, and the outcome window across reboots
  (`kdv.py:708-775`). Per-minute timestamps are deliberately NOT persisted (`rate_limiter.py:103`).
- **No-data warnings**: flags providers enabled > N hours with zero *attempts* (`kdv.py:225`).
- **Vendor cost recording**: `cost_adapter.record_vendor_cost` re-exports a DB writer for
  vendor adapters (`cost_adapter.py:11`).

## 2. Doesn't (explicit non-goals / boundaries)

From `record_failure` and module docstrings:

- **Not a model registry** — doesn't discover or enumerate models; host pushes config via
  `register()` (`kdv.py:98`).
- **Not a router / selector** — doesn't pick a model. It hands go/no-go + reliability signals
  to Fatih Hoca's selector via the nerd_herd adapter.
- **Not an LLM client** — makes no litellm/HTTP calls. HaLLederiz Kadir does that and *feeds*
  KDV headers/outcomes.
- **Not DaLLaMa** — no local-model or GPU management; the circuit breaker is cloud-only.
- **Ignores `auth` errors** — auth failures are permanent (credentials), not transient, so
  they do not trip the breaker and are excluded from the reliability window
  (`kdv.py:586`, `kdv.py:594`).
- **Hour bucket dropped** — Cerebras hour-axis headers are intentionally not tracked; the day
  bucket is sufficient to gate (`header_parser.py:339-348`).
- The DB *schema* for vendor cost is owned by `src/infra/db.py`; `cost_adapter` only owns the
  stable import contract (`cost_adapter.py:1-7`).

## 4. Edges (gotchas / footguns / couplings / contracts)

- **Bookkeeping site moved**: RPM is counted at `record_attempt`, NOT `post_call`
  (`kdv.py:159-186`, tests `test_kdv.py:77-104`). A caller that only calls `post_call` (no
  `record_attempt`) **undercounts RPM** and never increments attempt count → `no_data_warnings`
  will keep firing (`test_kdv_no_data_warning.py:46-61`). The canonical order is
  `pre_call → record_attempt → (post_call | record_failure+release_reservation)`.
- **Reservation contract**: if you pass `estimated_tokens` to `record_attempt`, you MUST pass
  the same value as `reserved_tokens` to `post_call` (success) OR call `release_reservation`
  (failure), or TPM accounting drifts (`kdv.py:204-223`, `kdv.py:381`).
- **Canary single-flight**: after ANY failure or at cold start, concurrent admissions on the
  same provider are refused with `reason="canary_in_flight"`, `wait_seconds=5.0`,
  `binding_provider=True` until one call returns (`kdv.py:289-296`). Callers that don't honor
  `reason` may busy-loop.
- **`wait_seconds` is load-bearing**: circuit-breaker refusals now return the real remaining
  cooldown, not `0.0` — a 2026-05-02 production bug where `wait=0.0` caused the dispatcher to
  skip its sleep gate and burst-reselect (`kdv.py:258-270`). `PreCallResult.reason` codes
  documented at `config.py:52-63` (`rpm`/`tpm`/`rpd`/`provider_*`/`circuit_breaker`/`canary_in_flight`).
- **Daily cooldown floor**: `record_failure(..., "daily_exhausted", msg)` floors the cooldown
  at 1 h and caps at 24 h regardless of the body's "retry in 29s" hint, because that hint is
  the immediate-retry, not the daily-window reset (`kdv.py:565-572`).
- **Header freshness / local projection**: `_header_rpm_remaining` is decremented locally on
  each `record_request` so the projection stays sharp between responses; validity is gated by
  `reset_at` or a 60 s fallback window (`rate_limiter.py:227-249`, `_decrement_header_rpm`
  :727). `is_rpm_cooldown` reads raw `_header_*` (bypasses the freshness window) because
  retry-after horizons exceed it (`rate_limiter.py:891`).
- **Restore is selective**: `RateLimitState.restore_state` skips `rpm/tpm/rpd/tpd` limits when
  `_rate_limit_hits == 0`, so freshly-registered seeds win over stale persisted limits
  (`rate_limiter.py:123-161`); a 2026-04-30 prod bug fix.
- **Callback safety**: `on_capacity_change` exceptions are caught and logged, never propagated
  (`kdv.py:93-96`).
- **In-flight push is OPT-IN**: until `configure_in_flight_push()` runs, `begin_call`/`end_call`
  track in-process only and never reach nerd_herd → `pressure_for()` sees `in_flight=0`
  (`__init__.py:33-55`). The push closure mutates the `CloudProviderState` it gets from the
  state_getter and writes both `rpm.in_flight` and `rpd.in_flight` (`in_flight.py:42-62`).
- **In-flight TTL**: handles auto-prune after `KDV_INFLIGHT_TTL_S` (default 180 s) if `end_call`
  never fires (crash safety); pruning is lazy on next `begin_call`/`count` (`in_flight.py:1-22, 64-69`).
- **Sync, single-process**: KDV itself is fully synchronous; atomicity of the check-and-reserve
  race relies on single-process asyncio (sync calls don't yield) (`kdv.py:163-167`). Async
  `wait_if_needed`/`wait_and_acquire` live on the rate limiter but are NOT used by the KDV
  facade or production caller path.
- **`auth_failure` vs `auth`**: `record_failure` ignores breaker tracking for `"auth"`
  (`kdv.py:586`) but excludes `"auth_failure"` from the reliability window (`kdv.py:594`) —
  two different string spellings, both handled, easy to confuse.

## 8. Nickname / persona

- **Correct spelling (with Turkish characters): "Küleden Dönen Var"** — note the dotted/dotless
  distinction: **küle** (with ü), **dönen** (with ö), **var**. The package/module slug is the
  ASCII transliteration `kuleden_donen_var`; the `__init__.py:1` docstring uses the proper form
  "Kuleden Dönen Var" (it carries the ö but writes "Ku" — the canonical persona name is
  **Küleden Dönen Var**).
- **Origin / meaning** (README lines 170, Turkish section): it is the Turkish phrasing of an
  **air-traffic-control tower announcement** — *"küleden dönen var"* ≈ "there's one cleared/
  returning from the tower," i.e. the control tower signals when a runway/slot opens up for an
  aircraft to land. The metaphor: KDV is the tower that tells the host application when a cloud
  provider's capacity has *opened up* (or closed), exactly as a tower clears aircraft. This maps
  directly onto the `on_capacity_change` callback firing `capacity_restored`.
- Note: literal Turkish "küle" means "ash"; the intended reading here is the **tower/landing-
  clearance** idiom per the README's own explanation, not a literal translation. (Flag for the
  rewrite: if the founder wants the idiom rendered precisely, confirm the exact ATC phrase — the
  README's gloss is the authoritative source for persona intent.)
