# i2p v3 Design Spec

## Goal

Create `i2p_v3.json` — a consolidated, difficulty-annotated workflow that routes hard steps to cloud models and keeps easy steps fast on local LLMs. Add engine support for three new per-step fields: `difficulty`, `tools_hint`, and `artifact_schema`.

## Constraints

- Steps must remain atomic — each doable by a small (7B) LLM with the right tools
- Difficult steps routed to cloud automatically via existing difficulty→ModelRequirements pipeline
- Engine is not battle-tested — high-reward rewrites are acceptable
- No cloud review loop changes (partially done, separate work)
- Backward compatible — v1/v2 workflows must still load and run

## Changes Overview

### 1. New Step Fields (JSON schema additions)

Every step in v3 gains three optional fields:

```json
{
  "id": "1.3",
  "phase": "phase_1",
  "name": "competitor_identification",
  "agent": "researcher",
  "difficulty": "easy",
  "tools_hint": ["web_search", "play_store", "github"],
  "artifact_schema": {
    "direct_competitors_list": {
      "type": "array",
      "min_items": 1,
      "item_fields": ["name", "url", "description", "category"]
    }
  },
  "depends_on": ["1.1"],
  "input_artifacts": ["idea_brief_final"],
  "output_artifacts": ["direct_competitors_list"],
  "instruction": "...",
  "done_when": "..."
}
```

#### 1a. `difficulty` field

Three levels mapping to existing numeric scale:

| Level | Numeric | Routing Effect | Use Case |
|-------|---------|---------------|----------|
| `"easy"` | 3 | Local small model, any available | Structured fill-in, single tool call, template-guided |
| `"medium"` | 6 | Local capable model or cloud if none loaded | Multi-tool, moderate synthesis, code generation |
| `"hard"` | 8 | Cloud preferred, or best local with thinking | Complex reasoning, multi-source synthesis, strategic decisions |

The expander maps these to `ModelRequirements.difficulty` in the task context. The existing router already handles difficulty-based model selection — no router changes needed.

Additionally, difficulty sets `needs_thinking` for hard steps (enables chain-of-thought on capable models) and `prefer_quality` (shifts scorer weights toward capability over cost).

#### 1b. `tools_hint` field

Array of tool names from TOOL_REGISTRY. Passed to the agent as `allowed_tools`, restricting which tools are available.

Benefits for small LLMs:
- No wasted iterations discovering tools via trial-and-error
- Focused tool descriptions in prompt = fewer tokens = more room for task context
- Prevents shopping agents from browsing filesystem, research agents from running shell commands, etc.

If omitted, falls back to agent type defaults (existing behavior).

#### 1c. `artifact_schema` field

Validation rules for output artifacts, checked in post-hook after step completion.

Schema types:
- `"object"` with `required_fields: ["field1", "field2"]` — output must be parseable JSON/dict with these keys
- `"array"` with `min_items: N` and optional `item_fields: [...]` — output must be parseable array with N+ items, each having listed fields
- `"string"` with `min_length: N` — output must be non-trivial text
- `"markdown"` with `required_sections: ["Section A", "Section B"]` — output must contain these headings

Validation flow:
1. Step completes → post-hook extracts output artifact
2. Validate against schema
3. If validation fails: retry step (up to 2 retries, agent gets validation error in prompt)
4. After 2 failed retries: accept best attempt, log warning, continue (don't block pipeline)

This is the single highest-ROI engine change — prevents garbage propagation from small LLMs.

### 3. Spike Phase (New)

Add step `8.spike` before feature implementation begins:

```json
{
  "id": "8.spike",
  "phase": "phase_8",
  "name": "feasibility_spike",
  "agent": "coder",
  "difficulty": "hard",
  "tools_hint": ["shell", "write_file", "read_file", "scaffold", "web_search"],
  "depends_on": ["7.last"],
  "input_artifacts": ["technical_design_document", "mvp_scope", "tech_stack_decision"],
  "output_artifacts": ["spike_report"],
  "artifact_schema": {
    "spike_report": {
      "type": "object",
      "required_fields": ["feasibility", "risks", "prototype_path", "recommendations"]
    }
  },
  "instruction": "Build a minimal throwaway prototype that validates the core technical risk. Focus on the hardest integration point (e.g., auth + DB + API for a single feature). Do NOT build the full app — just enough to prove the architecture works. Produce spike_report with: feasibility (go/no-go), risks discovered, prototype_path, and recommendations for the real implementation.",
  "done_when": "spike_report exists with feasibility assessment based on working prototype code."
}
```

2. **Phase 7→8**: Phase 7 setup artifacts (`repo_setup`, `ci_cd_pipeline_config`, `docker_setup`, `design_system_components`, `testing_infrastructure_setup`) wired as inputs to `8.0` (feature backlog init) and template context

### 5. Quality Gates in JSON (engine change)

Move gate definitions from hardcoded Python (`quality_gates.py`) into the workflow JSON:

```json
{
  "phase": "phase_9",
  "quality_gate": {
    "criteria": [
      {
        "name": "test_pass_rate",
        "artifact": "test_results",
        "check": "pass_rate >= 0.95",
        "message": "Test pass rate must be >= 95%"
      },
      {
        "name": "coverage",
        "artifact": "coverage_report",
        "check": "coverage >= 0.60",
        "message": "Code coverage must be >= 60%"
      }
    ],
    "on_failure": "needs_clarification"
  }
}
```

The engine reads these from the workflow JSON instead of hardcoded gate definitions. `quality_gates.py` becomes a generic evaluator that parses the `check` expression.

### 6. `skip_when` Field (engine change)

Steps can declare conditions under which they should be skipped:

```json
{
  "id": "14.12",
  "skip_when": ["no_mobile"],
  ...
}
```

At expansion time, the runner checks mission context for active skip conditions. Matching steps are inserted with `status=skipped` instead of `status=pending`.

Skip conditions are set from:
1. Mission context (user provides at workflow start)
2. Artifact-derived (e.g., if `monetization_strategy.pricing_model == 'free'`, activate `no_payment`)

This replaces some conditional groups with a simpler mechanism for binary skip/include decisions.

### 7. Engine Changes Summary

| File | Change | Reward |
|------|--------|--------|
| `expander.py` | Read `difficulty` → set in task context as numeric value + `needs_thinking` + `prefer_quality` for hard | High — enables cloud routing |
| `expander.py` | Read `tools_hint` → pass in task context | High — constrains small LLMs |
| `expander.py` | Read `skip_when` → skip steps matching mission context | Medium — reduces step count |
| `hooks.py` | Post-hook: validate output against `artifact_schema`, retry on failure | Highest — prevents garbage propagation |
| `hooks.py` | Post-hook: derive skip conditions from artifacts (e.g., free pricing → `no_payment`) | Medium — auto-skip |
| `base.py` | If task context has `tools_hint`, use as `allowed_tools` | High — tool focus |
| `quality_gates.py` | Rewrite to read gate definitions from workflow JSON instead of hardcoded | Medium — configurable gates |
| `runner.py` | Pass `skip_when` conditions, evaluate at expansion | Medium |
| `loader.py` | Validate new fields in step schema | Low — safety |

### 8. Difficulty Annotations (all steps)

Distribution target for v3:
- **Easy (~40%)**: ~76 steps — structured output, single tool, template-guided
- **Medium (~40%)**: ~76 steps — multi-tool, moderate synthesis, code generation
- **Hard (~20%)**: ~38 steps — complex reasoning, strategy, multi-source synthesis, code review

Hard steps (cloud-routed) include:
- All go/no-go decisions
- Architecture decisions (tech stack, patterns, schema design)
- Competitive analysis synthesis (not data collection)
- Code review steps
- Quality gate evaluations
- Security audit
- Spike/feasibility assessment
- Phase summary generation

### 10. Backward Compatibility

- `loader.py` treats `difficulty`, `tools_hint`, `artifact_schema`, `skip_when` as optional
- v1/v2 workflows load and run without these fields (existing behavior preserved)
- `version: "3.0"` in metadata distinguishes v3
- Old `quality_gates.py` hardcoded gates serve as fallback when workflow JSON has no gate definitions
