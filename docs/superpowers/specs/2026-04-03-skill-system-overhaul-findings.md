### Auto-captured Skills (139) — Garbage

Generated by `src/core/orchestrator.py:~2068-2076` after successful task execution.

**How auto-capture works**:
1. Task completes successfully (2+ iterations)
2. Orchestrator takes the task title, splits into words >= 3 chars
3. Joins words with `|` to create trigger_pattern
4. Saves with agent name, tools used, iteration count, model

**Why it produces garbage**:

1. **i2p step names become patterns** (132 of 139):
   - Task title: `[0.1] raw_idea_intake`
   - Generated pattern: `writer|[0.1]|raw_idea_intake`
   - `[0.1]` is invalid regex (character class matching `0`, `.`, `1`)
   - `raw_idea_intake` never appears in real user queries
   - These memorize workflow step IDs, not intent

2. **Task-specific, not generalizable** (7 non-i2p):
   - `coffee|machine` — only matches "coffee machine", not "laptop" or "vacuum"
   - `error|recovery:|task#1741` — matches literally one historical task
   - Learning "coffee machine → shopping_advisor" should generalize to "any product → shopping_advisor"

3. **Agent name pollutes pattern**: Patterns include `writer`, `analyst`, `researcher` etc. which are internal agent names, never in user queries.

### Injection Path (broken in 3 ways)

1. `skills.py:117` — `WHERE success_count > 0` excludes all seed skills
2. Auto-captured patterns don't match real queries (broken regex + title fragments)
3. `base.py:496` — exceptions caught at DEBUG level, silent empty returns

Even if injection worked, the auto-captured skills would rarely match because their patterns are i2p step names, not user intent keywords.

## What Needs Redesigning

### 1. Auto-capture must extract intent, not title words

Instead of splitting task titles into words, auto-capture should:
- Extract the **category/intent** of the task (shopping, research, coding, weather lookup, etc.)
- Use the task's **classifier output** or **agent type** to determine intent
- Build patterns from **semantic keywords** related to the intent, not the literal title
- For i2p tasks specifically: use the step's **purpose** (from i2p_v3.json description) not the step ID

### 2. Generalization needed

Current: "coffee machine" → `coffee|machine` pattern → only matches coffee machine queries
Needed: "coffee machine" was a shopping task → pattern should match shopping-intent queries broadly

Options to explore:
- **Category-based patterns**: Auto-capture creates/updates category-level skills (`shopping_product_search`) rather than instance-level (`coffee_machine_search`)
- **Vector-only matching**: Drop regex entirely for auto-skills, rely on ChromaDB semantic similarity with better descriptions
- **Hybrid**: Seed skills keep regex (they're hand-crafted), auto-skills use vector-only with rich descriptions

### 3. Prune the 139 garbage skills

The existing auto-captured skills should be cleaned up:
- Delete all 132 i2p step-name skills (broken regex, never match)
- Evaluate the 7 non-i2p skills — some may have useful tool_sequence metadata even if patterns are bad
- Or: wipe all auto-captured and let the redesigned system re-learn

### 4. Seed skills need success tracking

Seed skills are high-quality but have no usage data. Options:
- Give them `success_count = 1` at seed time (baseline trust)
- Track their usage separately (seed vs auto success rates)
- Remove the success_count filter entirely and rank by composite score

## Quick Wins vs Full Redesign

**Quick wins** (included in smart-resource-integration spec):
- Remove `WHERE success_count > 0` filter → seed skills start working
- `re.escape()` on auto-capture words → at least the regex is valid
- Upgrade logging to INFO/WARNING → failures visible

**Full redesign** (this doc, separate effort):
- Intent-based auto-capture
- Category generalization
- Prune garbage skills
- Better vector descriptions for semantic matching
- Possibly rethink what a "skill" even represents (routing hint? execution recipe? both?)

## DB Schema Reference

```sql
-- Current skills table
CREATE TABLE skills (
    name TEXT PRIMARY KEY,
    description TEXT,
    trigger_pattern TEXT,      -- regex pattern for matching
    tool_sequence TEXT,        -- JSON: agent, tools, iterations, model
    examples TEXT,             -- JSON array of example queries
    success_count INTEGER DEFAULT 0,
    failure_count INTEGER DEFAULT 0,
    created_at TEXT,
    updated_at TEXT
);
```

ChromaDB collection `skills` stores embeddings with metadata `{"type": "skill"}` for vector search fallback. Distance threshold: 0.7.
