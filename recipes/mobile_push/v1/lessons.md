# mobile_push Recipe v1 — Known Lessons

Pitfalls captured from prior implementations. Seeded into `mission_lessons` on recipe instantiation under `lessons_domain: mobile_push`.

- **Expo push tokens require a physical device**: `getExpoPushTokenAsync` throws or returns nothing on a simulator/emulator. `Device.isDevice` must be checked first — the shim does. Mission QA on a simulator will never produce a token; this is not a failure to "fix".
- **`projectId` is mandatory on SDK 49+**: `getExpoPushTokenAsync({projectId})` needs the EAS project id from `app.json` → `extra.eas.projectId`. Forget `eas init` and the call throws `"No projectId found"`. Wire it once at project setup.
- **iOS shows the permission prompt exactly once**: if the user declines `requestPermissionsAsync`, you cannot re-prompt programmatically — they must enable it in Settings. Always show your own rationale screen BEFORE the system prompt; a cold prompt with no context has a much higher decline rate.
- **Android 8+ drops channel-less notifications silently**: a push with no notification channel never appears and logs nothing. `setNotificationChannelAsync` must run before the first notification — putting it inside the registration function guarantees ordering.
- **Foreground notifications are silent without a handler**: `Notifications.setNotificationHandler` must be set (at module load, not in an effect) or notifications received while the app is open show nothing. SDK 50+ uses `shouldShowBanner`/`shouldShowList`, not the deprecated `shouldShowAlert`.
- **The backend talks to Expo, never to APNs/FCM**: for hosted push you POST to `https://exp.host/--/api/v2/push/send`. Wiring APNs `.p8` keys or an FCM server key into the backend is the self-managed path and conflicts with hosted push. Pick one.
- **Reject raw FCM/APNs tokens at registration**: a device may hand you a raw FCM token if the Expo modules are misconfigured. The Expo push API silently no-ops on a non-`ExponentPushToken[...]` value. Validate the prefix at `/push/register` (the stub does) so the bug surfaces immediately.
- **Evict tokens on `DeviceNotRegistered`**: an Expo *receipt* (fetched a few seconds after send) with `status:"error"` and `details.error == "DeviceNotRegistered"` means the user uninstalled or revoked. Keep pushing to it and Expo will eventually rate-limit you. Delete the row.
- **Tickets ≠ receipts**: the immediate `POST /push/send` response is a *ticket* (accepted-for-delivery). Real delivery errors arrive later via the *receipts* endpoint (`/push/getReceipts`). A 200 with an ok ticket does not mean the notification was delivered.
- **In-memory token store loses everything on restart**: `backend.template.py` uses a dict — a stub. Replace with a `push_tokens(user_id, expo_push_token UNIQUE, platform, updated_at)` table. Upsert on the token so a device that re-registers doesn't create duplicates.
- **`expo-notifications` is removed from Expo Go (SDK 53+) for Android remote push**: remote push testing increasingly requires a development build, not Expo Go. If QA can't get a token in Expo Go, build a dev client — this is an Expo platform change, not a recipe bug.
- **Badge counts need explicit management**: `shouldSetBadge` only controls whether the incoming push sets the badge. Clearing it (`setBadgeCountAsync(0)`) on app focus is your responsibility — stale badge counts are a common complaint.
