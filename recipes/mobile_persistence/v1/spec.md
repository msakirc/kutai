# mobile_persistence Recipe v1 — Specification

## Scope

On-device (frontend-only) data persistence for Expo apps. Ships a complete,
typed local-storage layer: a Drizzle schema, an `expo-sqlite` connection +
migration bootstrap, a query/mutation layer, and a `@tanstack/react-query`-
friendly data-hook layer — plus a `useLiveQuery` push-based hook and a simple
read-through cache.

There is **no backend** in this recipe. It is the persistence layer for the
mobile client. It pairs with — but does not require — a server recipe.

## When to Pick This Recipe

**Use** `mobile_persistence/v1` when:
- The app is built with **Expo** (`tech_stack` contains `expo`).
- You need to store **structured / relational data on the device** —
  anything you list, sort, paginate, filter, or relate.
- You want a typed query layer (Drizzle) instead of raw SQL strings.
- You want offline-first reads with optional server sync layered on later.

**Do NOT use** `mobile_persistence/v1` when:
- You only need a few flat settings (theme flag, onboarding-seen boolean) →
  use `AsyncStorage` directly, no recipe needed.
- You need cross-platform **web** persistence in the same codebase →
  `expo-sqlite` has no web fallback by default; use a different store for web.
- You need full bidirectional **offline sync** with conflict resolution →
  that is the stretch recipe `mobile_offline_sync` (Z5 T4), which builds on
  this one.
- The data is purely server-owned and never read offline → just call the API.

## What It Implements

Generic, single-entity (`<<ENTITY>>`) persistence — parameterized via
RECIPE_PARAM, not tied to any one feature:

| File | Role |
|------|------|
| `db/schema.ts` | Drizzle table definition — the single source of truth |
| `db/client.ts` | `expo-sqlite` handle + Drizzle instance + `runMigrations()` |
| `db/queries.ts` | Typed CRUD helpers + in-memory read-through cache |
| `db/hooks.ts` | react-query hooks + `useLiveNotes` (Drizzle live query) |
| `db/drizzle.config.ts` | `drizzle-kit generate` configuration |
| `migrations/0000_initial.sql` | Baseline migration (regenerated by drizzle-kit) |
| `migrations/migrations.js` | JS bundle the on-device migrator consumes |
| `frontend.template.tsx` | `PersistenceProvider` + live-query list screen |
| `tests/data_layer_smoke.ts` | Insert/read/update/delete + cache round-trip |

## Architecture

```
schema.ts  ──(drizzle-kit generate)──>  migrations/*.sql + migrations.js
   │                                          │
   ▼                                          ▼
client.ts  ── openDatabaseSync ──> Drizzle ── runMigrations() (app start)
   │
   ▼
queries.ts  ── typed CRUD + read-through cache (non-React call sites)
   │
   ▼
hooks.ts    ── react-query hooks  +  useLiveQuery (push-based)
   │
   ▼
frontend.template.tsx ── PersistenceProvider gate + list screen
```

## Key Design Choices

### `openDatabaseSync`, not `openDatabaseAsync`

Drizzle's Expo driver is built on the synchronous `SQLiteDatabase` handle.
The async handle throws at `drizzle()` construction. The file-open is cheap;
query execution through Drizzle stays async.

### Migrations gate the UI

`PersistenceProvider` awaits `runMigrations()` before mounting children. This
eliminates the most common crash class — querying a table before its
migration has run (`no such table`).

### Two consumption paths, one query layer

`queries.ts` is the only module that touches Drizzle. UI code uses the
react-query hooks (loading/error states, optimistic updates). Background /
non-React code uses `listNotesCached()` with the simple read-through cache.
They are not stacked for the same screen.

### Epoch-millis timestamps

SQLite has no native datetime type. Timestamps are `INTEGER` epoch-millis —
they sort and compare correctly and are timezone-free. ISO strings are never
stored (a recurring KutAI-wide pitfall).

## Read-through Cache

`listNotesCached()` returns a cached snapshot when fresh (< `CACHE_STALE_MS`),
else reloads from SQLite. Every mutation calls `invalidateCache()`. This is
the fallback for non-React call sites; for UI, react-query's `staleTime`
(mirroring `CACHE_STALE_MS`) does the same job with shared cross-screen cache.

## RECIPE_PARAM Markers

| Marker | Default | Description |
|--------|---------|-------------|
| `ENTITY` | `note` | Singular entity name |
| `ENTITY_PLURAL` | `notes` | Plural form — table + query-key naming |
| `TABLE_NAME` | `notes` | SQLite table name |
| `DB_FILE_NAME` | `app.db` | On-device SQLite file name |
| `CACHE_STALE_MS` | `30000` | Read-through cache / react-query staleTime window |

## Dependencies

**Frontend (npm)**:
- `expo-sqlite` — on-device SQLite database
- `drizzle-orm` — typed query builder + Expo driver + `useLiveQuery`
- `drizzle-kit` — migration generator (dev dependency)
- `@tanstack/react-query` — data hooks, caching, mutations

**Backend**: none — this is a frontend-only recipe.

## Post-hooks

`imports_check`, `test_run`, `pattern_lint` — run after the recipe's code is
emitted. (No `migration_apply`: migrations are applied on-device at app
start by the Expo migrator, not by a server-side migration step.)

## Known Non-Goals (v1)

- No web/SSR persistence fallback — `expo-sqlite` is native-only by default.
- No bidirectional server sync / conflict resolution → `mobile_offline_sync`.
- No encryption-at-rest (SQLCipher) — the SQLite file relies on OS-level app
  sandboxing; add SQLCipher in a v2 if storing sensitive data.
- No multi-table relations / migrations beyond the single `<<ENTITY>>` table —
  extend `schema.ts` and re-generate for richer schemas.
- No automatic data export/backup — the file is wiped on app uninstall.
