# mobile_offline_sync Recipe v1 — Known Lessons

Pitfalls captured from prior implementations. Seeded into `mission_lessons` on recipe instantiation under `lessons_domain: mobile_offline_sync`.

- **The queue MUST be durable, not in-memory**: a mutation queued in a JS array or React state is lost the instant the app is killed — exactly when offline users background the app. Persist to `expo-sqlite` (the recipe does). AsyncStorage holding a JSON blob also works but rewrites the whole blob on every enqueue and corrupts under concurrent writes.
- **Replay in insertion order**: drain by `id ASC`. Out-of-order replay breaks dependent mutations — replaying an `update item 7` before the `create item 7` that is also queued produces a 404 and data loss. The sync engine orders by id.
- **Mutations must be idempotent server-side**: a replay can hit a server that already processed the request (the original success response was lost on a flaky network). Use a client-generated id + an upsert, or a dedup key, so a double-replay is a no-op — never a duplicate row.
- **`networkMode: "offlineFirst"` on react-query**: without it, queries sit in `loading` forever when offline instead of serving cached data, and mutations are paused so the sync engine never sees them. Set it on BOTH `queries` and `mutations` defaults.
- **`isInternetReachable` can be `null` and lies**: `expo-network` reports a connected Wi-Fi even behind a captive portal with no real internet. Treat connectivity as a hint to *attempt* a drain; let replay failures re-queue. Never assume "connected" means "the API is reachable".
- **Cap retries — a poison mutation kills the queue**: a mutation the server permanently rejects (a 4xx) will retry forever and block everything behind it if you only delete-on-success. `bumpAttempts` + `MAX_REPLAY_ATTEMPTS` skips exhausted rows. Surface those rows to the user — silently dropping them is data loss they never consented to.
- **Use the async expo-sqlite API**: `openDatabaseAsync` / `runAsync` / `getAllAsync`. The legacy `openDatabase` callback API is deprecated and its transaction semantics differ. `runAsync` exposes `lastInsertRowId` — that is the queue row id you return to the caller.
- **Drain on mount, not just on reconnect**: mutations queued in a prior session are stranded if `drainQueue` only runs on a connectivity-change event (which never fires if the app launches already-online). `useOfflineSync` drains once on mount too.
- **Optimistic updates need reconciliation**: when a mutation is queued, the UI shows the optimistic state. After a successful drain, reconcile the react-query cache with the server's authoritative response — otherwise a server-side transform (e.g. a normalised field) is invisible until a full refetch.
- **A drain can race itself**: if `drainQueue` is slow and a reconnect event fires again mid-drain, two drains can replay the same row. Either guard with an in-flight flag or rely on server idempotency (preferred — see above). `dequeueMutation` after success keeps the window small but not zero.
- **expo-sqlite has no cross-process locking concern on mobile, but does within the app**: concurrent `runAsync` calls are serialised by the driver, but interleaving a `DELETE` (dequeue) with a `SELECT` (pending) can skip a row. Drain in a single sequential loop — the sync engine does — rather than `Promise.all` over the batch.
- **Batch the drain**: replaying thousands of queued rows in one tick blocks the JS thread and can trip API rate limits. `SYNC_BATCH_SIZE` caps each pass; call `drainQueue` again while `remaining > 0`.
- **This is replay, not merge**: two devices editing the same record offline will last-write-win on replay — the later drain overwrites the earlier. If the product needs true conflict resolution, that is a CRDT layer, explicitly out of scope for v1.
