## Substrate invariants SP3 MUST honor

Each came from a real shipped fix. Violating any one re-opens a silent-failure class.

- **Fire on DB `tasks.status` post-apply, never on in-memory `result["status"]`.** The substrate already gates this in `on_task_finished` (after `apply_actions`). Resume handlers receive the agent's pre-post-hook result snapshot. SP3 handler design depends on this — don't try to fire/re-route from inside post-hooks.
- **Handler signature is `async (task_id: int, result: dict, state: dict) -> None`.** `task_id` is the CHILD id. `state` is whatever you passed as `cont_state` at enqueue. Detached dispatch — handler errors are logged + swallowed.
- **`cont_state` must be JSON-serializable.** Round-trips through `state_json`. Carry parent_id, source_task_id, mission_id, whatever the resume needs to re-enter routing. Don't put non-serializable objects in there.
- **Empty-string `resume_name` = on_error-only continuation.** If you only need `on_error`, omit `on_complete`; `add_task` writes `""` as the sentinel.
- **`await_inline` and `on_complete`/`on_error` are mutually exclusive** — `enqueue` raises `ValueError`. Pick one.
- **`needs_clarification` leaves the row pending.** It is NOT terminal; the eventual `completed`/`failed` fires the continuation. Design your handlers accordingly.
- **Every new handler module MUST be added to `_HANDLER_MODULES`** in `packages/general_beckman/src/general_beckman/continuations.py`. SP1.1 made this a screaming docstring. Forgetting means: handler absent after restart → reconcile leaves rows pending → silent-but-recoverable, but operators see WARNING logs.
- **Handler-presence pre-check is on by default.** If your module's `register_continuations()` isn't called at startup (because you forgot `_HANDLER_MODULES`), `fire_for_task` won't claim — the row stays pending. Helpful in dev, but don't rely on it in prod.
- **Reconcile reconstruction is TOP-LEVEL JSON decode only.** Nested decoding (e.g. raw_dispatch's `{"content": "<JSON string>"}`) lives inside the handler. The reconcile docstring pins this no-magical-unwrap policy.

## Known traps (calibrated from SP1 ship pain)

- **Don't fire-then-mark.** The substrate's `claim_for_fire` is CAS-then-read. Don't introduce a new path that dispatches before claiming.
- **Don't await the handler.** Detached `asyncio.create_task` is the discipline. Slow handlers will stall the terminal pipeline otherwise.
- **Don't put `add_task` and `continuations` writes on different connections.** SP1 already routes both through `add_task`'s aux connection. New SP3 producers must use `enqueue(..., on_complete=..., cont_state=...)` — never write the continuations row directly.
- **Pre-existing flake to ignore:** `tests/beckman/test_enqueue_contract.py::test_enqueue_await_inline_blocks_until_resolved` flakes on cold-import (50ms `asyncio.sleep` vs. embedding-model first-load). Confirmed failing on base — SP5 deletes the test.
- **Pre-existing fail unrelated:** `packages/mr_roboto/tests/test_reversibility_registry.py::test_every_dispatcher_action_is_in_registry` complains about `publish_preview_pages`. Not yours.
